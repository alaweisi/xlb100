# XLB 三端 UI 生产章程

状态：`ACTIVE`
生效日期：2026-07-17
适用范围：`apps/customer`、`apps/worker`、`apps/admin`、`packages/ui` 以及对应测试与视觉证据。

## 1. 生产目标

XLB 三端 UI 不是静态 Demo，也不是独立于后端的画面集合。每条纵向切片必须把角色、商业目标、后端权威状态、权限、可执行动作、持久结果、异常恢复和跨端交接落实到真实 App。

范围基线固定为：

- 顾客端 62 条；
- 师傅端 54 条；
- 后台 98 条；
- 合计 214 条；
- 36 个 Carrier；
- 104 个顶层 Base/Gate/State Frame。

## 2. 四道不可替代的门禁

### G1 全中文界面

- 面向用户的标题、说明、标签、按钮、错误、空状态、恢复动作和无障碍名称必须使用中文；
- `API`、`SKU`、`ID`、`CNY` 等必要缩写只能在中文语境中出现；
- 后端枚举不得直接充当用户文案，必须经过中文显示映射；
- 门禁扫描三端 TSX 的可见文本，并保存文件、行号和原文。

### G2 214 条全量追踪

- `FRAME_MAP_SLICE_BINDINGS.md` 与机器总账必须一一对应；
- 214 条不得遗漏、重复或擅自新增；
- 每条切片状态只能按 `DEFINED → READY → IMPLEMENTED → API_INTEGRATED → TESTED → EDGE_VERIFIED → ACCEPTED` 前进；
- 只有 `ACCEPTED` 计入正式完成数量。

### G3 全量效果可查看

- 验收控制台必须索引全部 214 条切片和 36 个 Base Frame；
- 证据必须来自 Edge 中运行的真实三端 App；
- Region、Overlay、状态组件和微状态可以共享所属顶层画面，但每个 Slice ID 必须能定位到自己的操作与证据；
- 没有截图和进入步骤的画面保持未完成，不使用占位图。

### G4 真实商业 App

每条切片必须登记并验证：

- 真实 App 路由与源文件；
- 后端权威状态、角色权限和城市范围；
- 查询或写入所使用的真实 API binding；
- 操作后的持久结果；
- 失败、冲突、重复、离线和结果未知时的恢复动作；
- 下一责任人与跨端交接；
- staging seed 或 contract state 场景；
- 自动化测试文件。

静态按钮、硬编码成功、虚构经营数据、独立 Demo 页面和只在截图里存在的状态不能通过门禁。

## 3. 状态定义

| 状态 | 可验证条件 |
| --- | --- |
| `DEFINED` | Slice ID、Carrier 和设计落点已登记。 |
| `READY` | 后端事实、进入条件、权限和验收场景已确定。 |
| `IMPLEMENTED` | 真实三端页面和交互已实现。 |
| `API_INTEGRATED` | 查询/动作接入真实 API，结果写回持久 UI。 |
| `TESTED` | 相关单元、契约或 E2E 测试通过。 |
| `EDGE_VERIFIED` | Edge 中完成进入、操作、结果和异常画面验收。 |
| `ACCEPTED` | 四道门禁均通过，并记录验收人和时间。 |

不得跳过中间事实直接写入 `ACCEPTED`。机器门禁会重新检查必要字段和文件是否存在。

## 4. 人与 Codex 的责任

Human 负责正式服务类目、商业规则、优先级、高风险授权和发布决定。Codex 负责读取事实源、生成批次、实现、测试、截图、更新总账和报告缺口。

缺少关键业务事实时，对应切片保持阻断；不得为了凑齐画面发明价格、成功数据、权限、服务类目或 Provider 结果。

## 5. 完成声明

任何批次交付必须报告：

1. 本批次 Slice ID；
2. 真实页面入口；
3. API binding；
4. 自动化测试结果；
5. Edge 证据；
6. 总账状态变化；
7. 剩余阻断项；
8. 四道门禁结果。

“页面已做”“效果不错”“测试基本通过”均不是完成声明。
